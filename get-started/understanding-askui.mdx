## 2.1 How an AI Agent Works

An AI agent is a program that runs a continuous loop of three steps:

<Steps>
  <Step title="Perceive" icon="eye">
    Take a screenshot of the screen.
  </Step>
  <Step title="Decide" icon="brain">
    A vision-language model analyzes the screenshot and picks the next action.
  </Step>
  <Step title="Act" icon="hand-pointer">
    [AgentOS](/06-agent-os/index) executes it — click, type, scroll, press a key.
  </Step>
</Steps>

Then the loop repeats. The agent takes a fresh screenshot, sees what changed on screen, decides the next action, and continues — until the task is complete.

### Why this loop matters for multi-step workflows

Assume a multi-step workflow was executed through a single perceive-decide-act loop — the agent takes one screenshot at the start, plans all 10 steps, and executes them in sequence without looking at the screen again. If step 3 goes slightly wrong — a dialog appears in an unexpected position, a page loads slower than expected — every subsequent step runs against a screen that no longer matches what the agent assumed. Errors compound. By step 8, the automation is clicking in the wrong place entirely.

AskUI's agent avoids this because it runs the loop **once per step** — a fresh screenshot before every single action. If step 3 produces an unexpected result, the agent sees the actual state of the screen at step 4 and adapts. Each step is a self-contained perceive-decide-act cycle. This is what makes AI agents resilient across multi-step workflows.

In AskUI, three components make this loop work:

- **[AgentOS](/06-agent-os/index)** — the runtime on your machine that captures screenshots, moves the mouse, and types on the keyboard. It handles the *Perceive* and *Act* steps.
- **A vision-language model** (e.g. Claude Sonnet) — the AI that looks at each screenshot and decides what to do next. It handles the *Decide* step.
- **The [AskUI SDK](https://github.com/askui/python-sdk)** (`ComputerAgent`) — the Python code that orchestrates the loop. It calls AgentOS for screenshots, sends them to the model, interprets the model's response, and tells AgentOS to act.

## 2.2 The Demo Project — What's Inside

Open your file explorer and navigate to the `AskUI-Demo-Project` folder on your Desktop (inside `askui-projects`). You can look at every file directly — they're all plain text. Here's what each piece does:

```
AskUI-Demo-Project/
├── main.py              — The entry point. Reads tests, starts the agent, saves reports.
├── requirements.txt     — Python libraries the project needs.
├── .env.template        — Template for creating your credentials file.
├── .env                 — Your credentials (created in Chapter 1).
├── tests/               — Test definitions the agent will read and execute.
│   ├── rules.md         — Global rules that cascade to all tests.
│   └── …subfolders…     — Group related tests; each can have its own rules, setup, teardown.
├── prompts/             — Markdown files that define the agent's behavior and reporting style.
├── helpers/             — Custom tools the agent can use.
│   ├── get_tools.py     — Registers custom tools for the agent.
│   └── tools/           — Individual tool implementations.
├── plans/               — Plan files for selectively running a subset of tests.
└── agent_workspace/     — Reports and screenshots (created on first run).
```

## 2.3 Writing Tasks for AI Agents

Open the `tests/` folder in your file explorer and look at the files inside.

The key insight: **you don't write code to automate. You write instructions in plain language, and the AI figures out how to execute them.**

### Principle 1: Tell the agent *what* to do, not *how* to do it

You don't need to describe mouse coordinates, UI element locations, or pixel-level instructions. The agent sees the screen and figures out the mechanics. Your job is to state the goal clearly.

<Warning>
**Too much "how":**
Move the mouse to coordinates (340, 520) and left-click on the grey button that says "5". Then move the mouse 60 pixels to the right and click the orange "+" button.
</Warning>

<Check>
**The right level of "what":**
Open the Calculator application.
Calculate 5 + 3 using the on-screen buttons.
Verify the result displayed is 8.
</Check>

### Principle 2: Make verification its own step

When you need to check that something worked, write the verification as a separate step — not as a tag-on to the previous step. The agent takes a fresh screenshot for each step, so the verification sees the actual current state of the screen independently.

<Check>
**Verification as a separate goal:**
**Step 3:** Click the "=" button.
**Step 4:** Verify the display shows 8.
</Check>

### Principle 3: Keep steps atomic

Each step should do one thing. If you bundle multiple actions into a single step ("Open the browser, go to google.com, and search for AskUI"), the agent may succeed at all three, or fail partway through with no clear indication of where things went wrong. Atomic steps make runs easier to debug and easier to re-run selectively.

### Supported task formats

The demo project accepts tasks in five formats. Use whichever fits your workflow:

| Format | Best for | When to use |
| --- | --- | --- |
| `.txt` | Quick, informal tasks | You just want to type a few instructions and run them. No structure needed. |
| `.md` | Structured tasks with context | You want to specify an objective, steps, and deliverables. Gives the agent more context about *why* it's doing each step. |
| `.csv` | Tabular test cases | You have multiple test steps with preconditions and expected results — the format QA teams already use daily. |
| `.json` | Programmatic / pipeline-generated tasks | Tasks are generated by another tool, or need machine-readable IDs and step arrays for integration. |
| `.pdf` | Existing documentation or test specs | You already have test cases written up in a PDF — feed them directly without copying them out. |

Open the files in `tests/` to see working examples of each format. The CSV example is a particularly good starting point for QA professionals — it maps directly to the step/expected-result structure already used in most test case spreadsheets.

## 2.4 Rules, Setup, and Teardown

Each test folder can contain special files that control the agent's behavior for all tests in that folder:

| File | Purpose | When it runs |
| --- | --- | --- |
| `rules.md` | Context and constraints injected into the agent's prompt for every test in the folder | Always active |
| `setup.md` | Preparation steps — e.g. "Open Settings, enable WiFi" | Before any tests run |
| `teardown.md` | Cleanup steps — e.g. "Close all apps, return to home screen" | After all tests finish |

Rules cascade from parent to child folders. The root `tests/rules.md` applies globally to every test in the project. If you create a subfolder `tests/checkout-flow/` and add a `rules.md` inside it, that folder inherits the root rules *plus* its own — they stack.

## 2.5 How Prompts Shape the Agent

The files in `prompts/` are Markdown documents that get combined into the agent's **system prompt** — the set of instructions that tells the AI model how to behave, what it can do, and how to format its output.

Open each one in your file explorer and read through them. You'll find files describing:

- **System capabilities** — tells the agent what it can do: click, type, take screenshots, read files, write reports.
- **Device information** — tells the agent about the machine it's controlling: OS, screen resolution, available applications. The agent populates this automatically for each run.
- **Report format** — tells the agent how to structure its output: sections to include, how to reference screenshots, what counts as a pass or fail.

You can edit any of these to change agent behavior. For example, you could add a rule like "Always take a screenshot before and after each step" or "If a step fails, retry it once before marking it as failed."
